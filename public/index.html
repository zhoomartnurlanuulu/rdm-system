Зачем это нужно

Воспроизводимость как основа науки

Исследования всё больше опираются на данные. Без профессионального управления данные теряются, результаты невозможно воспроизвести, а вложенные инвестиции пропадают впустую.

"Growth in a Time of Debt" — ошибка в Excel, где пропустили 5 строк, изменила выводы с −0.1% до +2.2% роста ВВП.
Reinhart & Rogoff, 2010 — классический пример нарушения воспроизводимости
"Данные — это новая нефть, но только если правильно управлять ими."
The Economist, May 6, 2017

Цель системы — снять нагрузку с исследователей и обеспечить доверенную среду для сбора, хранения, анализа и публикации данных.

Принципы FAIR

Findable · Accessible · Interoperable · Reusable

Принципы FAIR — стандарт для машиночитаемых научных данных, принятый в EOSC (European Open Science Cloud) и поддерживаемый всеми ведущими фондами.

F
Findable
Данные и метаданные должны быть легко находимы как для людей, так и для машин.
  • Глобальный уникальный идентификатор (DOI)
  • Богатые метаданные для описания данных
  • Метаданные содержат идентификатор данных
  • Регистрация в поисковых индексах
Средний балл КГТУ
A
Accessible
Данные и метаданные должны быть доступны через стандартизированные протоколы.
  • Данные доступны по идентификатору через открытый протокол
  • Протокол обеспечивает аутентификацию при необходимости
  • Метаданные доступны даже без самих данных
  • REST API / OAI-PMH
Средний балл КГТУ
I
Interoperable
Данные должны использовать формальные, доступные, разделяемые и широко применимые языки для представления знаний.
  • Формальные, доступные, разделяемые онтологии
  • Словари FAIR в метаданных
  • Квалифицированные ссылки на другие данные
  • Wikidata, OWL, schema.org
Средний балл КГТУ
R
Reusable
Данные и метаданные должны быть хорошо описаны, чтобы их можно было повторно использовать в разных контекстах.
  • Множество точных атрибутов метаданных
  • Чёткая и доступная лицензия (CC-BY, CC0)
  • Подробное происхождение данных
  • Соответствие стандартам домена
Средний балл КГТУ
Invenio Framework

Core Principles of InvenioRDM

InvenioRDM is built on proven principles from CERN and partner institutions.

🔒

Safety & Long-term Preservation

Security and digital preservation are built into the core. Fixity checks, BagIt archival export, DOI minting and version linking guarantee data integrity for decades.

📈

Scalability by Design

Engineered for 100M+ records and petabytes of files. Distributed OpenSearch and PostgreSQL underpin linear horizontal scaling.

🌐

Open Standards & Open Source

MIT-licensed, JSON-native REST APIs, integration with ORCID, DataCite and OpenAIRE. Open science infrastructure any institution can deploy and extend.

✨

Excellent User Experience

UX is a first-class concern. Every feature goes through Balsamiq mockups, bi-weekly community review and RFC approval before any code is written.

🎯

Simplicity over Feature Bloat

"Design for power through simplicity." InvenioRDM deliberately avoids adding every requested feature. A focused interface serves researchers better.

🤝

Community-Driven Governance

Open, Inclusive, Considerate, Respectful. Decisions are made via RFCs with contributions from CERN, Northwestern University and institutions worldwide.

🏗️

Design-First Development

Ideas are discussed, real user needs gathered, mockups and RFCs written before implementation. This prevents costly rework and ensures intentional UX patterns.

🚀

Production Readiness

Every feature ships with tests, i18n, accessibility, docs and migration scripts. v13 powers Zenodo, CERN Repository, TU Graz and TU Wien in production.

Source: inveniosoftware.org  ·  InvenioRDM Development Process

Repositum — Публикационная база данных

DSpace + DOI Service · ПРОДАКШН

Репозиторий научных публикаций университета. Основан на платформе DSpace — ведущей open-source системе для институциональных репозиториев, используемой более чем 2000 организациями по всему миру.

  • Хранение статей, тезисов, книг, диссертаций
  • Присвоение постоянных идентификаторов DOI
  • Интеграция с австрийским DOI Service (адаптируется для КГТУ)
  • OAI-PMH для агрегации метаданных
  • Полнотекстовый поиск по всему репозиторию
  • ORCID-интеграция для авторов
● Продакшн

  • Git-репозитории для кода и конфигураций

TU-RDM / Invenio — Файловый репозиторий данных

Репозиторий для файловых данных: CSV, Excel, JSON, аудио, видео, изображения, геоданные. Каждый датасет получает DOI, лицензию и машиночитаемые метаданные.

  • Версионирование датасетов
  • JSON-метаданные по schema.org для машинного чтения
  • REST API для программного доступа

DBRepo — Репозиторий структурированных данных

MariaDB + Kubernetes + Docker · ПРОДАКШН

Репозиторий для структурированных данных в реляционных базах данных. Ключевая инновация — версионирование данных и цитирование произвольных подмножеств (Query Store + DOI).

  • Хранение структурированных данных без дампов/экспортов
  • Версионирование каждой записи (valid_from / valid_to)
  • Цитирование произвольных запросов через Query Store + DOI
  • REST API, Java API, Python API, JDBC
  • Message Queue (AMQP/MQTT) для IoT и потоковых данных
  • Семантическое картирование через Wikidata + OWL онтологии
  • Grafana-дашборды (автоматически генерируемые)
● Продакшн
-- Версионирование данных в DBRepo -- Каждое обновление не перезаписывает, а добавляет версию CREATE TABLE measurements ( id SERIAL PRIMARY KEY, station_id VARCHAR(50), parameter VARCHAR(20), value FLOAT, unit VARCHAR(10), -- Версионирование: version INT NOT NULL DEFAULT 1, valid_from TIMESTAMP DEFAULT CURRENT_TIMESTAMP, valid_to TIMESTAMP DEFAULT '9999-12-31' ); -- Получить актуальные данные: SELECT * FROM measurements WHERE valid_to = '9999-12-31'; -- Получить данные на конкретную дату: SELECT * FROM measurements WHERE valid_from <= '2024-06-01' AND valid_to > '2024-06-01';
DBRepo: ifs.tuwien.ac.at/infrastructures/dbrepo Demo: dbrepo1.ec.tuwien.ac.at IJDC Paper: doi.org/10.2218/ijdc.v17i1.825

DAMAP — Планирование управления данными

Java Spring Boot + React · ПРОДАКШН

Инструмент для создания машиночитаемых планов управления данными (maDMP). Автоматически предзаполняет данные из информационных систем университета, экспортирует для фондов Horizon Europe и Science Europe.

  • Пошаговое создание DMP по Science Europe Practical Guide
  • Предзаполнение из HR-базы, проектной БД, оргструктуры
  • Single Sign-On интеграция
  • Экспорт в Word для Horizon Europe, Science Europe
  • Машиночитаемый формат (maDMP) для автоматизации
  • Кастомизируемый внешний вид (логотипы, цвета)

OSSDIP — Безопасный доступ к чувствительным данным

VNC + VM + Мониторинг · ПИЛОТ

Инновационная концепция "Data Visiting" вместо "Data Sharing" для чувствительных и конфиденциальных данных. Исследователь приходит к данным, а не данные к исследователю.

  • Data Visiting: доступ без физического копирования данных
  • Временная виртуальная машина для каждого сеанса
  • Владелец данных контролирует: кому, когда, какое подмножество
  • Полный мониторинг всех действий в сессии
  • VNC-подключение к защищённой VM
  • Автоматическое уничтожение VM после сессии
◌ Пилот

# Поиск и скачивание данных через RDM API import requests, json BASE = "http://localhost:3000/api" KEY = "ваш-api-ключ" # из Личного кабинета → API # 1. Поиск датасетов по теме r = requests.get(f"{BASE}/datasets", params={ "q": "Иссык-Куль", "access": "open", "page": 1, "limit": 5 }) data = r.json() print(f"Найдено: {data['total']} датасетов") # 2. Скачать первый результат с API-ключом ds_id = data['items'][0]['id'] dl = requests.get( f"{BASE}/datasets/{ds_id}/download", headers={"Authorization": f"Bearer {KEY}"} ) with open("data.csv", "wb") as f: f.write(dl.content) # 3. Проверить FAIR-оценку fair = requests.get(f"{BASE}/datasets/{ds_id}/fair").json() print(fair['fair']) # {'F':75,'A':80,'I':60,'R':70}

Метаданные и требования финансирующих организаций

RDM КГТУ поддерживает ведущие стандарты метаданных и соответствует требованиям Horizon Europe, Science Europe и мировых репозиториев.

DC
Dublin Core
15 базовых элементов метаданных. Используется в Repositum (DSpace) для публикаций. OAI-PMH совместимый — агрегация в BASE, CORE, OpenDOAR.
dc.title · dc.creator · dc.identifier · dc.rights
DC4
DataCite Metadata Schema 4.x
Расширенная схема для наборов данных с DOI. Поддерживает ORCID, типы ресурсов, геопокрытие, источники финансирования и связанные идентификаторы.
resourceType · geoLocation · funder · relatedIds
SO
schema.org / JSON-LD
Машиночитаемые метаданные для поисковых систем. Google Dataset Search и OpenAIRE индексируют датасеты с этой разметкой. Выдаётся по эндпоинту /metadata.
@type Dataset · variableMeasured · distribution
DMP
maDMP — RDA DMP Common Standard
Машиночитаемые планы управления данными. JSON-LD формат с экспортом для Horizon Europe, Science Europe. Встроен в вкладку DMP личного кабинета.
dmp.dataset.distribution.license · contact.orcid

О системе

RDM КГТУ — по модели TU Wien

Система разработана на основе опыта Венского технического университета (TU Wien), который с 2018 года создавал профессиональную RDM-инфраструктуру. В 2024 году в TU Wien насчитывается более 300 датасетов в репозитории Invenio и работает Центр управления исследовательскими данными (ZFDM).

Ключевой принцип: ни одна система не удовлетворяет всем требованиям. Поэтому используется композиция специализированных сервисов, интегрированных через API Gateway.

Ключевые уроки из подхода TU Wien

01
Используйте композицию сервисов, а не один монолит
02
Начинайте с малого и постепенно усложняйте
03
Вовлекайте исследователей на всех этапах
04
Автоматизация критична для принятия системы
05
Доверие строится на прозрачности и индикаторах качества
06
Политика и оргподдержка так же важны, как технологии
